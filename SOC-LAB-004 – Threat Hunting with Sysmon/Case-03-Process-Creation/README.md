# Case Study 3 – Process Creation Investigation

## Scenario

During routine security monitoring, Wazuh generated an alert indicating the execution of a Windows command associated with user enumeration.

The alert was triggered after the creation of a new process involving the native Windows utilities **net.exe** and **net1.exe**.

The objective of this investigation was to determine whether the observed process creation represented legitimate administrative activity or potential attacker reconnaissance.

## Initial Evidence

Wazuh detected the creation of a Windows process associated with account enumeration.

The generated event identified the following process:

**Process**

```text
net1.exe
```

**Executed Command**

```cmd
net user
```

The **net user** command enumerates local user accounts on a Windows system.

Although this command is frequently executed by administrators, it is also commonly used by attackers during the **Discovery** phase after gaining access to a compromised endpoint.

For this reason, Wazuh generated an alert for investigation.

## Process Analysis

The investigation confirmed that the Windows utility **net.exe** launched **net1.exe**, which subsequently executed the `net user` command.

This parent-child relationship is expected Windows behavior. The **net.exe** utility internally invokes **net1.exe** to perform several administrative operations.

The executed command enumerates local user accounts, which explains why Wazuh mapped the activity to **MITRE ATT&CK T1087 – Account Discovery**.

Although the command is commonly used by attackers during post-exploitation, it is also a legitimate administrative command. Therefore, additional context was required before determining whether the activity represented a security incident.

## Analyst Assessment

The investigation confirmed that the activity was intentionally generated by the legitimate user **Marcio-Braga\yukem** during a controlled SOC laboratory exercise.

The parent-child process relationship matched the expected Windows behavior:

```text
net.exe
        │
        └── net1.exe
```

Both executables were launched from the legitimate **C:\Windows\System32** directory.

No evidence of persistence, privilege escalation, lateral movement, malicious payload execution, or suspicious child processes was identified during the investigation.

Although the command is frequently associated with attacker reconnaissance, the surrounding context confirmed that the activity was legitimate.

## 🧠 Analyst's Thought Process

> **Why was this alert generated?**
>
> Wazuh detected the execution of the Windows command `net user`, which is commonly associated with the **Discovery** phase of the MITRE ATT&CK framework. Attackers frequently use this command to enumerate local user accounts after gaining access to a compromised system.
>
> **What evidence was collected?**
>
> The investigation analyzed the executed command, process image, parent process, process hierarchy, user context, integrity level, process identifiers, cryptographic hashes, detection rule, severity level, and MITRE ATT&CK mapping.
>
> **What was investigated?**
>
> The investigation focused on validating the parent-child relationship (`net.exe → net1.exe`), the execution path, user context, command line, integrity level, and overall execution context to determine whether the activity represented normal Windows behavior or attacker reconnaissance.
>
> **What led to the final decision?**
>
> The process hierarchy matched the expected Windows behavior, both executables were launched from the legitimate **System32** directory, and the activity was intentionally generated during a controlled laboratory exercise.
>
> **Final Decision**
>
> The alert was classified as a **Benign True Positive**. Wazuh correctly detected a technique commonly associated with attacker reconnaissance; however, the surrounding context confirmed that the activity was legitimate.

## Lessons Learned

Windows administrative utilities are frequently abused by attackers because they are native operating system binaries and do not require additional tools.

However, the execution of these commands alone is insufficient to classify an event as malicious.

SOC analysts should always:

- Analyze the process hierarchy.
- Validate the parent process.
- Review the execution path.
- Inspect the command line.
- Verify the user context.
- Correlate additional Sysmon events.
- Evaluate the overall execution context.

Only after correlating these elements can the activity be accurately classified.

---

## Key Takeaway

> **Process creation alerts identify behavior—not intent.**
>
> The responsibility of a SOC analyst is to investigate the complete execution context, validate the parent-child relationship, correlate all available evidence, and determine whether the observed activity represents legitimate administrative behavior or a genuine security incident.
